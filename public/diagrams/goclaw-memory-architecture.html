GoClaw Memory & Knowledge Architecture

Three-layer intelligence system: per-user memory, knowledge graph, and session history — all PostgreSQL-backed with pgvector

overview Three Subsystems

GoClaw's intelligence layer is built on three complementary subsystems, each scoped per-agent and per-user for multi-tenant isolation.

Memory
Documents + chunks + embeddings
Hybrid search: pgvector + tsvector
Knowledge Graph
Entities + relations + traversal
LLM-extracted from conversations
Sessions
Full message history + metadata
Compaction + memory flush cycle

1 Memory System

Data Model

Three-tier storage: documents → chunks → embeddings. Every document is scoped to (agent_id, user_id) for tenant isolation.

TableKey ColumnsPurpose
memory_documents agent_id, user_id, path, content, hash Full markdown files (e.g., MEMORY.md, memory/2026-03-22.md)
memory_chunks document_id, text, embedding vector(1536), tsv tsvector Paragraph-level chunks with both vector embedding and full-text search index
embedding_cache hash, provider, model, embedding Deduplication cache — same text+model never re-embedded

Hybrid Search Pipeline

Every memory_search query runs two searches in parallel, then fuses results with configurable weights.

1
Vector Search — Query is embedded via EmbeddingProvider (OpenAI-compatible API, default text-embedding-3-small). Cosine similarity against memory_chunks.embedding using HNSW index.
2
Full-Text Search — PostgreSQL tsvector with 'simple' config (no stemming — works for Vietnamese and non-English). GIN-indexed tsv column on every chunk.
3
Rank Fusion — Results merged by VectorWeight and TextWeight (per-agent configurable). Filtered by MinScore threshold. Returns path, line range, score, snippet, and scope (global vs personal).

Multi-Tenant Scoping

Every memory operation is scoped by (agent_id, user_id). Agents see only their own memories. Users within an agent see only their personal memories + the agent's global memories (user_id IS NULL).

agent_id: UUID → agents(id) user_id: per-channel identity global scope: user_id = NULL personal scope: user_id = 'telegram:12345'

2 Knowledge Graph

Entity-Relation Model

Classic property graph stored in PostgreSQL (not Neo4j). Entities and relations are scoped per (agent_id, user_id).

kg_entities
external_idCanonical lowercase ID (e.g., "john-doe")
nameDisplay name in original language
entity_typeperson | project | task | event | concept | location | organization
description1-sentence summary
confidence0.0–1.0 (extraction certainty)
propertiesJSONB for arbitrary metadata
kg_relations
source_entity_idFK → kg_entities(id)
relation_typeworks_on, manages, depends_on, uses, etc.
target_entity_idFK → kg_entities(id)
confidenceExtraction confidence score
Dedup constraint: UNIQUE(agent_id, user_id, source, type, target) — same relation is never stored twice.

LLM Extraction Pipeline

Entities and relations are extracted from conversation text by an LLM call with a structured prompt.

1
Chunking — Long text split at paragraph boundaries (max 12,000 chars per chunk). Each chunk extracted independently.
2
LLM Extraction — System prompt defines 7 entity types and 15 relation types. LLM returns structured JSON with external_id, name, entity_type, confidence. Temperature=0, max_tokens=8192.
3
Filtering + Merging — Entities below minConfidence (default 0.75) are dropped. Cross-chunk dedup by external_id, keeping higher confidence. Relations deduped by (source, type, target).
4
Upsert — IngestExtraction() bulk-upserts entities and relations into PostgreSQL. Existing entities updated, new ones created.

Graph Operations

Search

Text search on entity name and description. Optional entity_type filter. Returns entities with their relations resolved to human-readable names. query="*" lists all entities.

Traversal

BFS from a start entity up to maxDepth (default 2, max 3). Returns connected entities with depth, path, and connecting relation type. Used for multi-hop relationship discovery.

Pruning: PruneByConfidence() removes entities and relations below a threshold — garbage collection for low-quality extractions.

Compaction + Memory Flush Cycle

When a session approaches the context window limit, GoClaw runs a two-phase cycle to preserve knowledge before truncating history.

1
Memory Flush (Pre-Compaction) — Sends flush prompt to LLM with file tools enabled. Agent writes durable memories to memory/*.md files. Max 5 tool iterations, 90s timeout. Increments memory_flush_compaction_count as dedup guard.
2
Extractive Fallback — If LLM returns NO_REPLY or fails, regex-based extraction kicks in. Captures: decisions ("decided to..."), user preferences ("I prefer..."), technical facts ("the API is..."), URLs, file paths, dates. Appends to memory/YYYY-MM-DD-auto-extract.md.
3
Compaction — History summarized into session.summary. Old messages truncated. Summary injected as context in future turns. compaction_count incremented.
4
Indexing — Flushed memory documents are chunked (paragraph boundaries), embedded via EmbeddingProvider, and stored with both vector(1536) and tsvector indexes for future hybrid search.

4 How The Three Systems Connect

Agent Loop Integration

During every agent turn, the three systems interact through the tool layer:

A
Recall — Agent calls memory_search tool → hybrid search across all memory documents. If knowledge graph is enabled, agent also calls knowledge_graph_search for entity/relationship queries. Results injected into context.
B
Conversation — Messages stored in sessions table as JSONB array. Token counts accumulated. Context window monitored for compaction threshold.
C
Persist — On compaction trigger: memory flush writes to memory_documents → chunks embedded → indexed. Knowledge graph extractor runs on flushed content → entities/relations upserted to kg_entities/kg_relations.
D
Context Files — Separate from memory: agent_context_files (shared across all users) and user_context_files (per-user). Injected into system prompt on every turn. Think AGENTS.md and USER.md.
