Most agent stacks still treat media as a side feature.
GoClaw treats it as a first-class execution path.

Image,
audio,
video,
and document ingestion are not bolted-on handlers.
They are organized as a tool family with shared routing patterns,
shared storage references,
and provider abstraction that can evolve without changing caller behavior.

The total implementation for media tools is large:
`internal/tools/create_*.go` and `read_*.go` contribute roughly 3,500 lines,
and `internal/tts/` adds another ~778 lines.
The result is a unified media subsystem spanning multiple vendors,
multiple payload shapes,
and multiple runtime behaviors.

Creation tools and read tools are intentionally paired.
One side generates assets.

At architecture level,

## <Label color="green">2</Label> Provider Chain Pattern (cyan)

<Card>

`media_provider_chain.go` is the routing core.
It chooses backend providers for creation tools,
applies fallback behavior,
and keeps dispatch logic centralized.

This is a strategy-chain architecture.
A tool request enters once,
and the chain selects a provider implementation that can satisfy the request.
If a provider is unavailable or unsuitable,
the chain can move to the next candidate according to configured policy.

The architectural win is decoupling.
Tool handlers do not need custom routing branches for every vendor.
Provider-specific logic stays in provider files,
while orchestration stays in the chain.

## <Label color="amber">3</Label> MediaRef: 60 Bytes vs Megabytes (blue)

<Card>

Large media payloads are expensive to pass inline,
especially in agent loops where context size is a hard limit.
GoClaw uses a MediaRef pattern to avoid this trap.

Instead of embedding full base64 media in every response,
tools persist media and return a lightweight reference object.
That object points to storage,
carries media type,
and keeps payload transfer separate from reasoning context.

The canonical shape is compact:
`ID`,
`MimeType`,
`Kind`,
`Path`.

That makes it feasible to pass media across tool boundaries,
chat channels,
and follow-up read operations,
without repeatedly inflating memory and token usage.

This is not just a storage optimization.
It is an architectural boundary.
The agent reasons about references,
while media bytes are handled by the delivery and persistence layers.

That boundary keeps long conversations healthy,
reduces repeated serialization cost,
and enables reuse of generated assets across subsequent tool calls.

## <Label color="rose">4</Label> Image Generation (violet)

<Card>

Image generation in GoClaw is implemented with provider-specific files,
including `create_image_dashscope.go` and `create_image_minimax.go`.

The APIs differ,
but the tool interface does not.
That means callers invoke `create_image` once,
and the provider chain handles request shaping,
invocation,
and response normalization.

This design protects upstream code from vendor drift.
A backend can change request parameters,
auth shape,
or response payload format,
without forcing changes in every agent workflow that consumes image generation.

Image output also benefits from MediaRef semantics.
Generated image bytes can be persisted,
returned as references,
and reused by read tools,
rather than repeatedly embedded in context.

Operationally,
this provides a stable image path for both synchronous agent tasks
and longer multi-step workflows where generated images are analyzed later.

</Card>

## <Label color="teal">5</Label> Audio Pipeline (purple)

Audio is where GoClaw combines breadth and structure.
`create_audio` supports multiple providers,
`read_audio` handles transcription,
and the dedicated TTS system in `internal/tts/` (about 778 lines)
provides a manager-based synthesis path across four providers.

The important point is coordination,
not just provider count.
Text input,
provider dispatch,
byte output,
and channel delivery are explicitly staged.

## <Label color="green">6</Label> Video Generation (fuchsia)

<Card>

Video generation is inherently asynchronous for many providers.
GoClaw models that reality directly.

With MiniMax and Veo-backed paths,
`create_video` can submit generation requests,
track job status,
poll for completion,
and normalize final outputs into the same reference-first response model.

`create_video_minimax.go` represents this async shape clearly:
request initiation and completion retrieval are separate concerns.
That allows the agent runtime to manage waiting semantics
without pretending video generation is immediate.

The architectural implication is important.
Asynchronous generation becomes a normal tool behavior,
not an exception path.
The media subsystem can therefore support long-running workloads
without leaking provider-specific state handling into callers.

Combined with MediaRef,
completed video assets are reusable across subsequent read operations,
summaries,
and downstream workflows.

</Card>

## The Strategy Pattern at Scale

GoClaw's media subsystem demonstrates strategy pattern discipline across modalities.
Each media type follows the same top-level dispatch model:
stable tool contract,
provider chain selection,
normalized output,
reference-based persistence.

That consistency is the scalability lever.
Adding a new provider is mostly a targeted adapter addition,
not a rewrite of caller contracts.

Across images,
audio,
and video,
the architecture repeats intentionally.
Repeated structure is what keeps a multi-provider system operable over time.

The result is a media pipeline that can grow with provider ecosystems
while keeping agent behavior predictable,
maintainable,
and cost-aware.

In short,
this is strategy pattern at production scale:
many backends,
one tool language,
and clean boundaries between orchestration and implementation.

That is why GoClaw can expand media capabilities quickly
without fragmenting the rest of the platform.
