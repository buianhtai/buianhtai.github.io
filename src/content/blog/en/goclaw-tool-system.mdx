When an LLM emits a tool call, most frameworks just... run it. GoClaw interposes an 8-step pipeline between intent and execution вҖ” policy checks, rate limiting, credential scrubbing вҖ” all before a single byte touches the filesystem. The tool registry isn't a convenience layer. It's the enforcement boundary.

## <Label color="green">2</Label> The 8-Step Execution Pipeline

<Card>

`ExecuteWithContext` is the only path from LLM intent to tool execution. Every tool call вҖ” filesystem, web, media, shell вҖ” passes through the same 8 steps. No shortcuts, no backdoors.

<Mermaid
  chart={`flowchart LR
    A[Tool Registration]
    B[Registry Discovery]
    C[Policy + Rate Limit]
    D[Argument Validation]
    E[Tool Execute]
    F[Scrub + Sanitize]
    G[Events + Return]

    A --> B --> C --> D --> E --> F --> G
    G --> H[Next LLM Iteration]
  `}
  caption="Tool lifecycle: register once, discover per call, execute through guarded pipeline"
/>

<FlowStep number={1} title="Registry Lookup">
Lock the registry mutex, resolve the tool by name from the registered map, unlock. Unknown tool names return an error immediately вҖ” the LLM gets feedback in the same iteration.
</FlowStep>

<FlowStep number={2} title="Policy Check">
The PolicyEngine evaluates the call against 5 layers: gateway defaults, global rules, agent-specific overrides, channel restrictions, and owner-level permissions. A single `deny` at any layer kills the call.
</FlowStep>

<FlowStep number={3} title="Rate Limiting">
Per-tool, per-agent, per-session rate limits via `rateLimiter.Allow(sessionKey)`. Prevents a runaway agent from hammering `web_fetch` 200 times in a single loop iteration.
</FlowStep>

<FlowStep number={4} title="Argument Validation">
Schema validation against the tool's `ToolDefinition` вҖ” required fields, type checks, enum constraints. Invalid arguments are rejected before execution, saving compute and preventing garbage-in/garbage-out.
</FlowStep>

<FlowStep number={5} title="Tool Execution">
`tool.Execute(ctx, args)` runs with a timeout derived from the tool's configuration. Context carries per-call values injected in steps 1-4: channel, chat ID, peer kind, sandbox key.
</FlowStep>

<FlowStep number={6} title="Credential Scrubbing">
`ScrubCredentials()` strips API keys, tokens, and passwords from both `ForLLM` and `ForUser` output fields. The LLM never sees raw credentials in tool results вҖ” even if the tool itself leaks them.
</FlowStep>

<FlowStep number={7} title="Result Sanitization">
Truncate oversized outputs, normalize response format, ensure consistent structure for the agent loop. Large file reads or verbose command outputs get trimmed to fit the context window.
</FlowStep>

<FlowStep number={8} title="Event Emission">
Emit `tool.call` and `tool.result` events for distributed tracing. Duration is logged вҖ” slow tools surface in metrics without any instrumentation inside the tool itself.
</FlowStep>

</Card>

<Callout color="green">
Context keys вҖ” not mutable struct fields вҖ” carry per-call values. This is what makes tool instances safely shareable across concurrent goroutines. One `read_file` instance serves every agent in the process.
</Callout>

This isn't fire-and-forget. Each goroutine gets its own context (channel, chat ID, sandbox key), runs through policy and rate limiting independently, and returns a typed result. The merge step preserves tool call ordering so the LLM sees results in the same sequence it requested them.

The key constraint: **parallel execution happens within a single LLM iteration**. Across iterations, execution is always sequential вҖ” the agent loop waits for all tools to complete before sending results back to the LLM.

## <Label color="rose">4</Label> Virtual FS Routing

<Card>

`read_file` and `write_file` don't go straight to disk. Three interceptor layers route specific paths to entirely different backends вҖ” the tool itself doesn't know the difference.

<FlowStep number={1} title="Context File Interceptor">
Intercepts 7 known filenames вҖ” `SOUL.md`, `IDENTITY.md`, `AGENTS.md`, `TOOLS.md`, `USER.md`, `BOOTSTRAP.md` вҖ” and routes them to PostgreSQL. Open agents get per-user context files; predefined agents share agent-level files with only `USER.md` per-user.
</FlowStep>

<FlowStep number={2} title="Memory Interceptor">
Routes `memory/*.md` paths to the memory document store. Writes automatically trigger `IndexDocument()` вҖ” chunking and embedding happen inline, so memories are searchable immediately after creation.
</FlowStep>

<FlowStep number={3} title="Sandbox / Host FS">
Everything else falls through to the actual filesystem. If sandboxing is enabled, this means a Docker container with its own isolated mount. Otherwise, `resolvePath()` joins with the workspace root and validates with `HasPrefix()` to block path traversal.
</FlowStep>

</Card>

<Callout color="rose">
7 context files are stored in PostgreSQL, not the filesystem вҖ” enabling per-user persistence without directory isolation.
</Callout>

## <Label color="green">5</Label> Credentialed CLI

<Card>

The `exec` tool runs shell commands. `credentialed_exec` does something fundamentally different вҖ” it runs binaries with injected credentials, using direct `exec` (no shell), so there's no command injection surface.

**How `credentialed_exec` works:**

1. **Credential lookup** вҖ” Binary name maps to encrypted environment variables in the `secure_cli_binaries` database table
2. **Shell operator detection** вҖ” Blocks `;`, `|`, `&&`, `||`, `>`, `<`, `$()`, backticks. If any are found, the call is rejected before execution
3. **Path verification** вҖ” Binary is resolved to an absolute path and matched against the configured path. A workspace-local `./gh` can't masquerade as the system `gh`
4. **Per-binary deny patterns** вҖ” Optional regex patterns block dangerous arguments (e.g., `auth\s+`, `ssh-key` for the `gh` binary)
5. **Direct exec** вҖ” `exec.CommandContext(binary, args...)` with credentials injected as environment variables. No shell involved, ever

**Available presets:** `gh`, `gcloud`, `aws`, `kubectl`, `terraform`

</Card>

<Callout color="green">
Credentials are injected via environment variables, never as command arguments. They're also registered with the scrubber вҖ” if a credential leaks into stdout, it gets redacted before the LLM sees it.
</Callout>
