DeepSeek streams thinking tokens in a dedicated field. [GoClaw](https://github.com/nextlevelbuilder/goclaw)'s provider layer absorbs all of this behind a four-method interface — the agent loop never has to care.

## <Label color="teal">1</Label> The Provider Interface

<Card>

This is the entire contract. Every LLM backend in GoClaw — from Anthropic's native HTTP+SSE to a local Ollama instance — implements these four methods.

```go
type Provider interface {
    // Chat sends messages and returns a complete response.
    Chat(ctx context.Context, req ChatRequest) (*ChatResponse, error)

    // ChatStream sends messages and streams response chunks via callback.
    // Returns the final complete response after streaming ends.
    ChatStream(ctx context.Context, req ChatRequest,
        onChunk func(StreamChunk)) (*ChatResponse, error)

    // DefaultModel returns the provider's default model name.
    DefaultModel() string

    // Name returns the provider identifier (e.g. "anthropic", "openai").
    Name() string
}
```

The `ChatRequest` carries messages, tool definitions, model override, and an options map for provider-specific settings like `temperature` or `thinking_level`. The `ChatResponse` unifies content, thinking output, tool calls, finish reason, and token usage — regardless of which backend produced it.

</Card>

<Callout color="teal">
Two methods to implement. That's the entire contract. Everything else — retry logic, schema cleaning, token counting — lives outside the interface.
</Callout>

## <Label color="amber">3</Label> Streaming vs Non-Streaming

<Card>

Both `Chat()` and `ChatStream()` return the same `ChatResponse` — the difference is *when* the caller sees data.

**`Chat()`** — Single request/response. The provider makes an HTTP POST, waits for the complete JSON body, parses it, and returns. Simple. Used when the caller doesn't need real-time output (sub-agent calls, background tasks).

**`ChatStream()`** — SSE stream with `onChunk` callback. The provider opens a streaming HTTP connection, reads server-sent events line by line, and fires the callback for each content delta. Internally it accumulates everything and returns the final assembled response when the stream ends.

```go
func (p *OpenAIProvider) ChatStream(
    ctx context.Context,
    req ChatRequest,
    onChunk func(StreamChunk),
) (*ChatResponse, error) {
    body := p.buildRequestBody(req, true) // stream: true
    resp, err := p.doRequest(ctx, body)   // HTTP POST
    if err != nil { return nil, err }
    defer resp.Body.Close()

    var accumulated string
    scanner := bufio.NewScanner(resp.Body) // SSE line reader
    for scanner.Scan() {
        line := scanner.Text()
        if !strings.HasPrefix(line, "data: ") { continue }

        chunk := parseSSEDelta(line)
        if chunk.Done { break }

        onChunk(StreamChunk{Content: chunk.Delta}) // Emit to caller
        accumulated += chunk.Delta
    }

    return &ChatResponse{Content: accumulated}, nil
}
```

</Card>

<Callout color="amber">
Key difference for retry: non-streaming wraps the entire request in `RetryDo`. Streaming retries only the connection phase — once SSE events start flowing, no retry occurs mid-stream.
</Callout>

## <Label color="rose">4</Label> Extended Thinking

<Card>

Not all providers support reasoning tokens. Some do — Anthropic has `budget_tokens`, OpenAI has `reasoning_effort`, DashScope has `thinking_budget`. GoClaw handles this through a separate `ThinkingCapable` interface that providers can optionally implement.

```go
// ThinkingCapable is optionally implemented by providers
// that support extended thinking / reasoning tokens.
type ThinkingCapable interface {
    SupportsThinking() bool
}
```

The agent loop checks at runtime:

```go
if tc, ok := provider.(providers.ThinkingCapable); ok && tc.SupportsThinking() {
    req.Options["enable_thinking"] = true
    req.Options["thinking_budget"] = agentConfig.ThinkingBudget
}
```

Thinking content streams separately from regular content. The `StreamChunk` struct has distinct `Content` and `Thinking` fields — the UI can render them differently (e.g., collapsible reasoning panel vs. main output).

```go
type StreamChunk struct {
    Content  string // Regular response content
    Thinking string // Extended thinking / reasoning
    Done     bool   // Stream complete
}
```

Each provider maps its native thinking format to this unified struct:
- **Anthropic**: `thinking` content blocks with `budget_tokens` config
- **OpenAI**: `reasoning_content` field with `reasoning_effort` (low/medium/high)
- **DashScope**: `thought` field with `thinking_budget` parameter
- **DeepSeek**: `reasoning_content` on the message object

</Card>

<Callout color="rose">
Not all providers support thinking. The `ThinkingCapable` interface lets the loop check at runtime without breaking non-thinking providers. It's Go's version of optional capabilities — no inheritance, just interface assertion.
</Callout>

## <Label color="green">5</Label> Retry Logic & Resilience

<Card>

Every provider call is wrapped in `RetryDo` — a generic function with exponential backoff, jitter, and context cancellation.

```go
func RetryDo[T any](
    ctx context.Context,
    cfg RetryConfig,
    fn  func() (T, error),
) (T, error) {
    for attempt := 0; attempt < cfg.Attempts; attempt++ {
        result, err := fn()
        if err == nil {
            return result, nil
        }
        if !IsRetryableError(err) {
            return result, err
        }

        // Exponential backoff with jitter
        delay := min(cfg.MinDelay * (1 << attempt), cfg.MaxDelay)
        jitter := time.Duration(rand.Int63n(int64(float64(delay) * cfg.Jitter)))
        delay += jitter

        select {
        case <-ctx.Done():
            return result, ctx.Err()
        case <-time.After(delay):
        }
    }
    return zero, fmt.Errorf("all %d attempts exhausted", cfg.Attempts)
}
```

The backoff schedule with default config (3 attempts, 300ms min, 30s max, 10% jitter):

```
Attempt 1: 300ms  ±30ms   → 270ms..330ms
Attempt 2: 600ms  ±60ms   → 540ms..660ms
Attempt 3: 1200ms ±120ms  → 1080ms..1320ms
```

## <Label color="teal">6</Label> Schema Cleaning

<Card>

Different providers accept different JSON Schema features in tool definitions. An agent writes one tool definition — the provider layer silently strips unsupported fields before sending them over the wire.

`CleanSchemaForProvider()` recursively walks the entire schema tree — including nested `properties`, `anyOf`, `oneOf`, and `allOf` — and removes fields the target provider rejects.

Schema cleaning is invisible to the agent — it writes one tool definition, and the provider layer adapts it for each backend. No conditional logic in agent code, ever.
</Callout>

## <Label color="rose">7</Label> What v2.0.0 Adds to Providers

<Card>

**Breaking: `type` → `driver`** — Provider configurations now use `driver` instead of `type` to specify the backend. Update your `goclaw.json` under `llm.providers` to rename the field.

**Prompt Caching** — Anthropic providers send explicit `cache_control` markers on system prompts and long conversation prefixes. OpenAI, MiniMax, and OpenRouter-compatible providers get automatic caching. Cache hit/miss metrics are tracked per-span in the trace viewer.

**New Providers** — v2.0.0 adds Claude CLI (Anthropic via stdio + MCP bridge) and Codex (OpenAI gpt-5.3-codex via OAuth with `TokenSource` interface). Both implement the same four-method `Provider` interface — the loop doesn't know the difference.

**Extended Response Types** — `ChatResponse` now carries a `Phase` field for Codex (distinguishing "commentary" from "final_answer") and `RawAssistantContent` for Anthropic (preserving thinking blocks through tool loops). `Usage` tracks `CacheCreationTokens`, `CacheReadTokens`, and `ThinkingTokens`.

**MediaRef** — A new lightweight type (~60 bytes) that replaces inline base64 images in session storage. Media files are persisted to disk and referenced by ID, keeping session JSONB small even with heavy image/video workflows.

</Card>
