The most visible parts of GoClaw are agent behavior and tool execution.

The most foundational part is the gateway.

It is where authentication is enforced, where policy hooks are evaluated, and where both WebSocket and REST clients converge on the same runtime methods.

This layer is what makes GoClaw usable by humans, automations, and external systems at the same time.

The architecture is intentionally unified.

Different access protocols are supported, but execution authority stays centralized.

That keeps policy, auditing, and behavior consistent across interfaces.

---

## <Label color="teal">1</Label> API Key Authentication (teal)

<Card>

GoClaw's gateway uses multi-key authentication with RBAC-aware scopes.

Keys are not stored as plaintext.

The system stores SHA-256 hashes of key material and compares hashed values during request authentication.

That design narrows the blast radius if storage is exposed.

Key creation follows a show-once pattern.

The full token is displayed once at creation time and is not retrievable later.

Operationally, this prevents accidental long-term key leakage through dashboards or logs.

Keys can also include optional expiration and revocation.

That gives teams practical lifecycle controls for machine credentials.

## <Label color="green">2</Label> RBAC Scopes (cyan)

<Card>

Authentication proves identity.

Authorization defines capability.

GoClaw scopes API keys across a hierarchy:

gateway-level,
agent-level,
and channel-level permissions.

This allows one key to be broad enough for infrastructure automation while another key is tightly constrained to one agent or channel workflow.

Scope granularity matters because gateway methods can perform powerful actions.

A flat all-or-nothing key model would force teams to over-provision access.

GoClaw avoids that by letting permissions match operational intent.

This hierarchy has two operational effects.

First, key blast radius can be minimized.

Second, audit trails are easier to interpret because capability boundaries are explicit in policy.

When a request is denied, teams can reason about where the policy boundary was crossed.

That is much easier than debugging a monolithic permission model.

## <Label color="amber">3</Label> Event-Driven Hooks (blue)

<Card>

Hooks let the gateway enforce policy at runtime events.

Events can include tool execution, message send paths, and other lifecycle points where external checks should run.

GoClaw supports two evaluator types.

The command evaluator runs a shell command and uses exit code semantics.

The agent evaluator delegates review logic to another agent.

Both can be configured as blocking gates.

That means an operation can be paused until hook evaluation returns an allow outcome.

Auto-retry support helps absorb transient failures.

One critical safety rule is recursion protection.

Hook evaluations do not trigger more hooks.

Without that guard, policy checks could recurse into infinite policy loops.

With that guard, hooks remain deterministic and bounded.

## <Label color="rose">4</Label> WebSocket RPC v3 (violet)

<Card>

The real-time dashboard is powered by WebSocket RPC v3.

The protocol handles connection handshake, authentication, subscriptions, chat streaming, and server-pushed events.

This is the transport that gives operators and users low-latency visibility into live system behavior.

A typical session flow is straightforward.

The client connects,
authenticates,
subscribes to channels or streams,
then receives both chat responses and gateway events over the same persistent connection.

## <Label color="green">6</Label> REST API (fuchsia)

<Card>

GoClaw also exposes REST endpoints for integrations.

Examples include `/v1/chat/completions` (OpenAI-compatible), `/v1/agents`, `/v1/skills`, and related resources.

Interactive API exploration is available at `/docs` via Swagger UI.

The REST layer is intentionally thin.

It wraps the same gateway methods used by WebSocket RPC.

That gives integrators standard HTTP ergonomics while preserving one core execution model behind the interface.

A thin wrapper architecture has concrete benefits:

- less duplicate business logic,
- consistent auth and hook enforcement,
- fewer mismatches between protocol surfaces,
- easier testability and maintenance.

## One Gateway, Multiple Faces

GoClaw intentionally supports three access patterns with one shared core:

WebSocket for real-time interaction,
REST for integration compatibility,
and API keys for controlled machine access.

Hooks enforce programmable policy at runtime,
RBAC scopes constrain capability,
and hashed key storage protects credentials at rest.

The result is a gateway layer that is flexible at the edges but consistent in execution.

Different clients can enter through different doors,
but they all meet the same method model,
the same policy checks,
and the same operational guarantees.
