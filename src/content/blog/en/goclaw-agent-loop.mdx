## <Label color="teal">1</Label> The Run() Entry Point

<Card>

`Run()` in `loop_run.go` handles the per-run lifecycle. Every agent invocation — whether it's a direct message, a sub-agent delegation, or a group chat participant — enters through this single function.

It manages three critical responsibilities: **atomic run counting** for concurrency control, **trace creation** for observability, and **event emission** so the host application can track agent activity in real time.

```go
func (l *Loop) Run(ctx context.Context, req RunRequest) (*RunResult, error) {
    l.activeRuns.Add(1)
    defer l.activeRuns.Add(-1)
    
    // Emit run started event
    emitRun(AgentEvent{Type: protocol.AgentEventRunStarted})
    
    // Create or attach trace (parent/child for delegation)
    traceID := l.createTrace(ctx, req)
    
    // Execute the core loop
    result, err := l.runLoop(ctx, req)
    
    // Emit completion with usage stats
    emitRun(AgentEvent{Type: protocol.AgentEventRunCompleted})
    return result, nil
}
```

</Card>

<Callout color="teal">
`activeRuns` uses an atomic counter — true concurrency for group chats with `maxConcurrent=3`.
</Callout>

## <Label color="green">2</Label> The 7-Phase runLoop()

<Card>

Every call to `runLoop()` executes these seven phases in order. Each phase has a single responsibility and clear failure semantics — if any phase fails, the run stops and returns a structured error.

<FlowStep number={1} title="Setup">
Inject agent UUID, user ID, per-user workspace (`sync.Map` cache), team workspace auto-resolution.
</FlowStep>

<FlowStep number={2} title="Input Validation">
`InputGuard.Scan` for 6 injection patterns (configurable: log/warn/block). Message truncation at 32K chars.
</FlowStep>

<FlowStep number={3} title="Build Messages">
System prompt (19 sections), conversation summary injection, 3-stage history pipeline (limit → prune → sanitize).
</FlowStep>

<FlowStep number={4} title="LLM Iteration Loop">
`PolicyEngine` filters available tools, calls `Chat` or `ChatStream`, mid-loop compaction at 75% context window.
</FlowStep>

<FlowStep number={5} title="Tool Execution">
Single tool: sequential. Multiple tools: parallel goroutines. Emit `tool.call`/`tool.result` events.
</FlowStep>

<FlowStep number={6} title="Response Finalization">
7-step sanitization, `NO_REPLY` detection, atomic session flush with deferred writes.
</FlowStep>

<FlowStep number={7} title="Auto-Summarization">
Trigger at 50+ messages or 75% context, per-session `TryLock` prevents concurrent summarization.
</FlowStep>

## <Label color="amber">3</Label> Think → Act → Observe

<Card>

The iteration loop is the heart of the agent. Each cycle follows the classic Think → Act → Observe pattern, but with a critical addition: **mid-loop compaction**. When the conversation approaches 75% of the context window, GoClaw summarizes the history *without stopping the loop*.

```go
for iteration := 0; iteration < maxIterations; iteration++ {
    // THINK: Call LLM
    response, err := l.callLLM(ctx, req, messages)
    
    // Mid-loop compaction check
    if shouldCompact(messages, l.contextWindow) {
        messages = l.compactHistory(ctx, messages)
    }
    
    // OBSERVE: No tool calls → done
    if len(response.ToolCalls) == 0 {
        return &RunResult{Content: response.Content}, nil
    }
    
    // ACT: Execute tools (parallel if multiple)
    for _, tc := range response.ToolCalls {
        result := l.executeTool(ctx, tc)
        messages = append(messages, toolResultMsg(result))
    }
    // Loop back to THINK
}
```

The loop terminates in three ways: the LLM returns a response without tool calls (natural completion), the iteration limit is reached (`maxIterations` default: 20), or an unrecoverable error occurs.

</Card>

<Callout color="amber">
Mid-loop compaction is the key insight — rather than failing when context overflows, GoClaw summarizes history mid-conversation and keeps going.
</Callout>

### The 3-Stage History Pipeline

Before messages reach the LLM, they pass through three processing stages that ensure the conversation fits within the context budget while maintaining coherence.

<FlowStep number={1} title="limitHistoryTurns">
Truncate to `historyLimit` — keeps only the N most recent conversation turns. Older turns are already captured in the conversation summary.
</FlowStep>

<FlowStep number={2} title="pruneContextMessages">
Respect context window budget — measures token count and removes oldest messages first until the total fits within the available context (after accounting for system prompt and response buffer).
</FlowStep>

<FlowStep number={3} title="sanitizeHistory">
Fix orphaned `tool_call`/`tool_result` pairs — if pruning removed a tool call but left its result (or vice versa), this stage removes the orphan to prevent LLM confusion.
</FlowStep>

## <Label color="green">5</Label> The Provider Interface

<Card>

Every LLM provider in GoClaw implements a single four-method interface. This is the entire contract — no base classes, no inheritance, no framework magic.

```go
type Provider interface {
    Chat(ctx context.Context, req ChatRequest) (*ChatResponse, error)
    ChatStream(ctx context.Context, req ChatRequest, onChunk func(StreamChunk)) (*ChatResponse, error)
    DefaultModel() string
    Name() string
}
```

The `Chat` method handles synchronous request/response. `ChatStream` handles server-sent events with a chunk callback. Both return the same `ChatResponse` struct — streaming implementations accumulate chunks internally and return the final assembled response.

</Card>

<Callout color="green">
One interface, 20+ implementations — Anthropic (native HTTP+SSE), OpenAI, Groq, DeepSeek, Gemini, Ollama, Claude CLI, and any OpenAI-compatible endpoint.
</Callout>
