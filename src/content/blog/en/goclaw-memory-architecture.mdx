Most AI agents are stateless. They forget everything the moment a conversation ends. GoClaw takes a different approach — it gives every agent a durable, searchable memory that survives across sessions, users, and even compaction cycles.

The system has three layers, all backed by PostgreSQL: a **Memory Store** (documents + chunks + embeddings with hybrid search), a **Knowledge Graph** (LLM-extracted entities and relations), and **Session History** (with compaction and memory flush cycles).

## Why This Architecture?

**One database for everything.** Documents, embeddings, knowledge graph, sessions, and context files — all in PostgreSQL. No Redis, no Neo4j, no Pinecone. pgvector handles similarity search. tsvector handles full-text. JSONB handles schema-flexible metadata. The operational complexity stays low.

The tradeoff is scale — this architecture works for thousands of agents with millions of memories, but not billions. For a self-hosted AI agent platform, that's the right tradeoff.

### Key Design Decisions

- **Hybrid search** (vector + full-text) outperforms either alone. The `'simple'` tsvector config works across languages including Vietnamese.
- **Knowledge graph in PostgreSQL** — no need for Neo4j when your graph queries are bounded to depth 3 BFS traversal.
- **Pre-compaction memory flush** ensures knowledge is never lost when sessions are truncated. The regex fallback catches what the LLM misses.
- **Embedding cache** by content hash eliminates redundant API calls — same text is never embedded twice.
- **Multi-tenant scoping** via `(agent_id, user_id)` on every table provides isolation without separate databases.
