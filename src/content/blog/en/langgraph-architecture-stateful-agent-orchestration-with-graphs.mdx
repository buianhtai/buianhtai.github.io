Most agent demos look convincing until they need to survive a retry, pause for human approval, or resume with state that outlives a single request. LangGraph is the layer built for that harder problem.

tools: Execute tools
review: Validate result
human: Human checkpoint
store: Persist state
done: Final answer

user -> router
router -> planner
planner -> tools: if external action
planner -> review: if enough context
tools -> store: checkpoint
store -> review: resume
review -> human: risky / ambiguous
human -> review: approve or edit state
review -> done

planner.shape: hexagon
review.shape: hexagon
store.shape: cylinder
human.shape: person

  <SlidePanel
    id="runtime"
    kicker="Slide 04"
    color="rose"
    title="A production LangGraph app is really four stacked layers"
    summary="State is the system of record, nodes do narrow work, routing controls topology, and the runtime layer makes the whole thing operable."
  >
    <div class="grid gap-6 xl:grid-cols-[0.95fr_1.05fr] xl:items-start">
      <div class="space-y-4">
        <FlowStep step="1" color="teal" title="State layer">
          Define the durable data model for the run: messages, planner outputs, tool results, retrieved context, flags,
          and review decisions.
        </FlowStep>

        <FlowStep step="2" color="green" title="Execution layer">
          Keep nodes narrow. Good nodes classify, plan, call tools, review output, or finalize. They should feel more
          like application steps than chat prompts.
        </FlowStep>

        <FlowStep step="3" color="amber" title="Routing layer">
          Edges, conditional edges, `Send`, and `Command` decide what happens next. This is the control plane, not an
          implementation detail.
        </FlowStep>

        <FlowStep step="4" color="rose" title="Durability layer">
          Persistence, interrupts, tracing, and deployment turn the graph from notebook logic into a runnable
          application runtime.
        </FlowStep>
      </div>

      <div class="prose prose-lg max-w-none">
        <p>
          This is why LangGraph feels different from higher-level agent abstractions. The center of gravity is not the
          prompt. It is the <strong>state machine around the prompt</strong>.
        </p>
        <p>
          That design changes how teams think about reliability. Instead of reconstructing history from logs, they can
          inspect the graph state directly, resume from checkpoints, and reason about which node was entered and why.
        </p>
        <Callout color="green">
          Prompts are inputs to nodes. State is the system of record. That is the architectural shift.
        </Callout>
        <p>
          Once you accept that model, many production concerns become natural: long-running threads, resumable work,
          human checkpoints, and topology changes that can be migrated safely over time.
        </p>
      </div>
    </div>

          <li>routing depends on typed intermediate state,</li>
          <li>humans may need to intervene,</li>
          <li>tasks pause, resume, or branch,</li>

        <h3>Skip it when</h3>
        <ul>
          <li>a single prompt plus one tool call solves the problem,</li>
          <li>there is no durable state to preserve,</li>
          <li>plain request-time code is enough,</li>
          <li>the team is still exploring the shape of the workflow.</li>
        </ul>
