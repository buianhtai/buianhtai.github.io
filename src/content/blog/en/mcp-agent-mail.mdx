Run three AI agents on the same codebase — backend, frontend, infrastructure — and watch them overwrite each other's files, lose context from parallel workstreams, and force you to manually relay messages between them. There's no shared memory, no coordination protocol, no audit trail.

[MCP Agent Mail](https://github.com/Dicklesworthstone/mcp_agent_mail) solves this by giving agents what humans have had for decades: **email**. Named identities, threaded conversations, inbox/outbox semantics, and advisory file leases — all exposed as MCP tools over HTTP, backed by SQLite for fast queries and Git for human-auditable history.

### Conflict Detection

```python
# Check for existing reservations on the same path
rows = conn.execute("""
  SELECT agent_name, path, exclusive, expires_ts
  FROM file_reservations
  WHERE project_id=? AND released_ts IS NULL
    AND expires_ts > ? AND path=?
""", (project_id, now, path)).fetchall()

# Conflict if anyone holds exclusive, or we want exclusive
if any(r["exclusive"] or exclusive for r in rows
       if r["agent_name"] != agent_name):
    conflicts.append({"path": path, "holders": [dict(r) for r in rows]})
```

Returns `{ "granted": [...], "conflicts": [...] }` — the agent decides how to proceed.

## <Label color="teal">6</Label> Concurrency & Reliability

Running multiple agents means concurrent database access. Agent Mail handles this with three patterns:

<Card>
### SQLite WAL + Exponential Backoff

```python
# WAL mode for concurrent reads during writes
cursor.execute("PRAGMA journal_mode=WAL")
cursor.execute("PRAGMA busy_timeout=60000")  # 60s

# Retry with exponential backoff + jitter
@retry_on_db_lock(max_retries=7, base_delay=0.05, max_delay=8.0)
async def get_session():
    # 0.05s → 0.1s → 0.2s → 0.4s → 0.8s → 1.6s → 3.2s
    delay = base_delay * (2 ** attempt)
    jitter = delay * 0.25 * random.uniform(-1, 1)
```

</Card>

<Card>
### Circuit Breaker

After 5 consecutive database failures, the circuit opens — failing fast instead of piling up retries:

```
CLOSED (normal) → 5 failures → OPEN (fail fast, 30s)
                                    ↓
                              HALF_OPEN (try one)
                                    ↓
                              success → CLOSED
```

<Mermaid
  chart={`flowchart LR
    C[CLOSED]
    O[OPEN (30s)]
    H[HALF_OPEN]

    C -->|5 failures| O
    O -->|30s elapsed| H
    H -->|success| C
    H -->|failure| O`}
  caption="Circuit breaker state machine used for database reliability under repeated failures"
/>

</Card>

<Card>
### Git Commit Batching

A `_CommitQueue` batches non-conflicting commits to reduce lock contention:

```python
class _CommitQueue:
    max_batch_size = 10   # Up to 10 commits per batch
    max_wait_ms = 50.0    # Wait up to 50ms for more commits
```

Under high load, 10 agent messages become 1 Git commit instead of 10 — reducing lock acquisition cycles by 90%.

</Card>

## What Makes It Work

The key design decisions behind MCP Agent Mail:

1. **Dual persistence** — SQLite for machine-speed queries, Git for human-auditable history
2. **Advisory leases, not hard locks** — agents signal intent without blocking each other
3. **Named identities** — GreenCastle is instantly recognizable in logs, diffs, and conversations
4. **Thread-based conversations** — `thread_id` preserves context across message chains
5. **Profile-based tool exposure** — reduce context overhead for constrained agents
6. **Commit batching** — 10 messages → 1 Git commit under load

It's not real-time pub/sub. It's not a shared database. It's **email** — the coordination protocol that scaled human organizations for 50 years, adapted for AI agents that work at machine speed.
