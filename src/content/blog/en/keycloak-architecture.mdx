Every non-trivial application needs authentication. You can wire up JWT validation, session management, password hashing, MFA, social login, SAML federation, and LDAP sync yourself — or you can deploy a system that's already handled those problems for CERN's 300,000 users, Japanese pharmaceutical networks with 2M+ patients, and Austrian government portals serving millions of businesses.

[Keycloak](https://github.com/keycloak/keycloak) is that system: an open-source identity and access management server, Apache 2.0 licensed, CNCF incubating, backed by Red Hat. It speaks OIDC, OAuth 2.0, and SAML 2.0 natively. It federates against LDAP/Active Directory out of the box. It handles SSO across dozens of applications without your users ever seeing a second login screen.

## <Label color="teal">1</Label> Why Keycloak Exists

Building authentication is deceptively complex. A login form is trivial. Enterprise-grade identity management — with SSO, token rotation, MFA, federation, fine-grained authorization, and regulatory compliance — is not.

<Card>

| Problem | DIY Cost | Keycloak Solution |
|---------|----------|-------------------|
| **SSO across apps** | Weeks of session/cookie engineering | Built-in, works with any OIDC/SAML client |
| **LDAP/AD federation** | Custom sync logic, password delegation | Native LDAP provider, configurable sync |
| **Social login** | OAuth client per provider, profile mapping | 20+ built-in identity providers |
| **MFA / Passkeys** | TOTP library + WebAuthn integration | Configurable authentication flows |
| **User management** | Admin UI, self-service, password policies | Full admin console + account console |
| **Token management** | JWT signing, refresh rotation, revocation | Complete OIDC token lifecycle |

Keycloak's vision follows the 80/20 rule: support 80% of use cases out of the box, and provide SPIs for the remaining 20% through custom extensions.

</Card>

## <Label color="teal">5</Label> The SPI Extension System

Keycloak's architecture is built around Service Provider Interfaces — 20+ extension points where you plug in custom behavior without forking the core:

<Card>

```java
// Example: Custom Authenticator
public class CaptchaAuthenticator implements Authenticator {

    @Override
    public void authenticate(AuthenticationFlowContext context) {
        // Show CAPTCHA challenge
        Response challenge = context.form()
            .setAttribute("captchaSiteKey", getSiteKey())
            .createForm("captcha-form.ftl");
        context.challenge(challenge);
    }

    @Override
    public void action(AuthenticationFlowContext context) {
        String captchaResponse = context.getHttpRequest()
            .getDecodedFormParameters().getFirst("captcha");
        if (verifyCaptcha(captchaResponse)) {
            context.success();
        } else {
            context.failureChallenge(AuthenticationFlowError.INVALID_CREDENTIALS,
                context.form().setError("captchaFailed").createForm("captcha-form.ftl"));
        }
    }
}
```

Deploy: package as JAR → copy to `providers/` → run `kc.sh build`. Zero core modification.

</Card>

Every SPI follows the same bootstrapping pattern — declare a `Spi` implementation that names the extension point, its provider interface, and its factory:

<Card>

```java
// How Keycloak discovers SPIs internally (ThemeSpi example)
public class ThemeSpi implements Spi {
    @Override
    public String getName() { return "theme"; }

    @Override
    public Class<? extends Provider> getProviderClass() {
        return ThemeProvider.class;
    }

    @Override
    public Class<? extends ProviderFactory> getProviderFactoryClass() {
        return ThemeProviderFactory.class;
    }
}
```

Concrete providers like `FolderThemeProvider` (filesystem) and `JarThemeProviderFactory` (classpath) implement this SPI. At startup, Quarkus's `KeycloakRecorder` discovers all SPIs and wires them into the `KeycloakSessionFactory` — a closed-world set with zero runtime scanning overhead.

</Card>

## <Label color="teal">9</Label> Getting Started

Three commands to a running identity server:

<FlowStep step="1" color="teal" title="Docker (development)">

```bash
docker run -p 8080:8080 \
  -e KC_BOOTSTRAP_ADMIN_USERNAME=admin \
  -e KC_BOOTSTRAP_ADMIN_PASSWORD=admin \
  quay.io/keycloak/keycloak:26.5 start-dev
```

Open `http://localhost:8080` → Admin Console → create a realm, client, and users.

</FlowStep>

<FlowStep step="2" color="green" title="Kubernetes (production)">

```yaml
# Install the Operator
kubectl apply -f https://raw.githubusercontent.com/keycloak/keycloak-k8s-resources/26.5/kubernetes/keycloaks.k8s.keycloak.org-v1.yml

# Deploy a Keycloak CR
apiVersion: k8s.keycloak.org/v2alpha1
kind: Keycloak
metadata:
  name: my-keycloak
spec:
  instances: 3
  db:
    vendor: postgres
    url: jdbc:postgresql://postgres:5432/keycloak
  hostname:
    hostname: auth.example.com
```

The Kubernetes Operator handles rolling upgrades, database migrations, and TLS configuration.

</FlowStep>

<FlowStep step="3" color="amber" title="Connect your app (Spring Boot example)">

```yaml
# application.yml — no Keycloak adapter needed
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://auth.example.com/realms/my-realm
```

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.authorizeHttpRequests(auth -> auth
            .requestMatchers("/public/**").permitAll()
            .requestMatchers("/admin/**").hasRole("ADMIN")
            .anyRequest().authenticated())
            .oauth2ResourceServer(oauth2 -> oauth2.jwt());
        return http.build();
    }
}
```

Standard Spring Security — no vendor-specific adapter. Works with any OIDC-compliant server.

</FlowStep>

## What Makes It Different

Keycloak isn't the newest or the flashiest IAM solution. It's the one that organizations choose when they need **protocol completeness, extension depth, and cost predictability at scale**.

1. **Protocol breadth** — OIDC + SAML + LDAP + SCIM + FIDO2 in a single server. Most alternatives lack native SAML brokering.
2. **SPI architecture** — 20+ extension points. Custom authenticators, protocol mappers, user storage providers — all deployable as JARs without forking.
3. **Enterprise federation** — LDAP/AD sync, Kerberos/SPNEGO, identity brokering with any SAML/OIDC provider. The glue between modern apps and legacy identity infrastructure.
4. **Zero per-user cost** — Scales to millions of users with only infrastructure costs. Auth0 pricing becomes prohibitive past 25K MAU.
5. **CNCF governance** — Community-driven, not controlled by a single vendor's roadmap. 1,350+ contributors, Red Hat backing for enterprise support.
6. **Kubernetes-native** — Operator-managed deployment, CRD-based configuration, rolling upgrades with zero downtime.

The project is a decade old, actively maintained with quarterly releases, and trusted by organizations from CERN to UnitedHealthcare. For teams with the operational expertise to run it, Keycloak is the most capable open-source identity server available.

Open source: [github.com/keycloak/keycloak](https://github.com/keycloak/keycloak) (33,500 stars, Apache 2.0, Java + TypeScript).
